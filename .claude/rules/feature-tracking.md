# Feature Tracking Protocol

## 开发前

每次开始一个新功能/调查/bugfix 之前：

1. **查找已有文档**：先检查 `docs/features/` 是否已有相关文档
2. **有文档 → 读完再动手**：文档里记录了之前的尝试、发现、失败原因、当前状态
3. **无文档 → 创建再动手**：按下面模板创建

## Feature 文档模板

路径：`docs/features/<YYYY-MM-DD>-<feature-name>.md`

```markdown
# Feature: <名称>

## 目标
一句话描述要实现什么

## 当前状态
🔴 未开始 / 🟡 进行中 / 🟢 已完成 / ⛔ 受阻

## 关键发现
<!-- 每次有新发现时追加，带时间戳 -->
- [HH:MM] 发现 XXX，因为 YYY
- [HH:MM] 测试 ZZZ，结果是 AAA

## 已尝试的方案
<!-- 记录失败的方案，避免重复 -->
| 方案 | 结果 | 原因 |
|---|---|---|
| ... | ❌ 失败 | ... |

## 当前方案
<!-- 正在执行或下一步要做的 -->

## 相关文件
<!-- 涉及的源码文件 -->

## 依赖/阻塞
<!-- 被什么阻塞，或阻塞了什么 -->
```

## 开发中

- **每个关键发现**立刻更新文档（不要等做完）
- **每次失败的尝试**记录方案 + 失败原因
- **切换方案时**更新"当前方案"和"当前状态"
- **被阻塞时**标记 ⛔ 并写清楚阻塞原因

## 开发后

- 更新状态为 🟢 已完成
- 确保"关键发现"和"已尝试的方案"完整
- 这个文档就是下次相关开发的起点

## 为什么这样做

- 上下文超限后新会话可以快速恢复
- 避免重复尝试已失败的方案
- 多 agent 并行时有共享的信息来源
- 渐进式披露：先看目标和状态，再看细节
